## 한 줄

모니터 없는 컨테이너에서 VIO 알고리즘이 `cannot open display`로 죽었다. 이게 **X11의 클라이언트-서버 구조** 문제임을 짚고, 없던 X 서버를 Xvfb로 세워 풀었다.

## 증상

헤드리스 서버/CI의 Docker 컨테이너에서 VIO를 돌리면 `cannot open display` 와 GTK/VTK/X11 assertion으로 크래시. 빌드는 멀쩡한데 **런타임에** 터진다. 문제 알고리즘이 셋, 각자 다른 GUI 라이브러리에 묶여 있었다.

- OKVIS2-X → GTK
- Kimera-VIO → VTK
- ORB-SLAM3 → Pangolin

## 어느 레이어인가 — 내 진단

처음엔 단순하게 생각했다. GUI 프로그램이 최종 영상을 화면 픽셀에 한 번에 차르륵 올리는 거라면, 모니터가 없을 때 그냥 출력이 안 될 뿐이지 왜 *연결 실패* 같은 에러가 날까?

에러 문구가 걸렸다. `cannot open display` — 파일이 없다도, 권한이 없다도 아니고, 마치 **어딘가에 연결을 시도했다 실패한** 모양새다. 그래서 픽셀에 직접 꽂는 게 아니라 *뭔가를 거친다*는 쪽으로 의심이 갔다.

생각을 밀어봤다.

> **내 진단** 데스크탑에서 프로그램 둘이 동시에 창을 띄우면 각자 화면 메모리에 마구 쓸 수 없다 — 누군가 화면을 독점해서 중재해야 한다. 그 중재자는 내 앱 *안*이 아니라 **바깥의 별도 존재**여야 모든 GUI 앱을 통솔할 수 있다. 여기서 그림이 맞춰졌다: **클라이언트-서버 구조**. 내 VIO 앱은 그 중재자에게 그림을 부탁하는 클라이언트고, `open display`는 화면 쇳덩이가 아니라 **그 서버로 연결을 여는 것**이었다.

## CS 개념

X11(리눅스 GUI의 토대)은 **client-server** 구조다.

- **X 서버(display server)** — 화면·입력장치를 독점 관리하는 별도 프로세스. 모든 그리기 요청을 중재한다.
- **X 클라이언트** — GUI 앱. 서버에 "이거 그려줘"를 보낸다. GTK·VTK·Pangolin·Qt 전부 결국 X 클라이언트다.
- 둘은 **IPC**(유닉스 소켓, 또는 TCP)로 대화한다.
- 클라이언트는 **`DISPLAY` 환경변수**로 "어느 서버에 붙을지"를 안다. `DISPLAY=:99` 면 99번 디스플레이의 소켓(`/tmp/.X11-unix/X99` 같은)으로 연결한다.

핵심은, 클라이언트 입장에선 *반대편에 X 서버가 존재하기만 하면* 된다는 것. 진짜 모니터가 달렸는지는 클라이언트가 알 바 아니다.

## 좁혀가기

여기서 한 끗을 잘못 짚을 뻔했다. "서버에 화면 사용권을 **요청했는데 거절당했다**"고 생각했는데 — 그러면 권한을 얻어내는 게 해결책이 된다.

다시 따져보니 아니었다. 헤드리스 컨테이너엔 *거절할 문지기 서버조차 떠 있지 않다.* 로컬 데스크탑엔 (서버용 우분투를 깐 게 아니면) X 서버가 늘 돌고 있지만, 컨테이너는 기본적으로 X 서버가 **없다.** 그러니 거절이 아니라 **연결할 상대가 아무도 없어서 연결 자체가 불발**된 것. 이 구분이 해결책의 방향을 가른다 — 권한을 얻어내느냐(X), 없는 서버를 세우느냐(O).

## 해결 & 왜 되는가

없던 서버를 직접 하나 세운다. **Xvfb(X virtual framebuffer)**:

```bash
apt install -y xvfb
Xvfb :99 &
export DISPLAY=:99
```

- `Xvfb`는 **진짜로 작동하는 X 서버**다. "휴지통에 버리는" 게 아니라, 픽셀을 실제 모니터 대신 **RAM 속 가상 프레임버퍼**에 그린다. 그림은 실제로 그려지고(원하면 스크린샷도 뜬다), 단지 볼 사람이 없을 뿐. VIO 앱의 GUI 초기화 코드는 *서버가 존재하기만* 하면 안 죽으니, 아무도 그 viewer를 볼 필요가 없다.
- `:99`는 **주소**다. `Xvfb :99`는 거기에 소켓을 열어 서버를 세우고, `DISPLAY=:99`는 클라이언트에게 "그 주소로 붙어라"라고 일러준다. 포트번호로 양쪽을 맞추는 것과 같다.

**왜 이게 세 알고리즘을 한 번에 덮는가** — 여기가 핵심이다. `DISPLAY`는 라이브러리마다 맞추는 게 아니라 **환경변수 하나**다. GTK·VTK·Pangolin 모두 *같은* `DISPLAY`를 읽어 서버를 찾는다. 그래서 Xvfb 한 대 + `export DISPLAY=:99` 한 번이면 세 알고리즘의 GUI 라이브러리가 전부 그 한 서버에 붙는다. 고치는 지점이 세 라이브러리 *아래의 공유 환경 레벨*이라 한 번으로 전부 해결된다.

### 대안: 의존성을 채우지 말고 없애기

반대 전략도 있다 — 서버를 세우는 대신, GUI 초기화가 *아예 돌지 않게* 만든다. ORB-SLAM3는 config에 `bUseViewer: false`라는 공식 스위치가 있다. 직접 소스의 viewer 초기화를 주석 처리할 수도 있다.

하지만 이건 라이브러리 *각각*에 손대야 한다 — 세 알고리즘이면 세 번, 게다가 각 소스를 **포크·패치로 떠안는** 셈이라 viewer가 다른 초기화 로직과 얽혀 있으면 깨지고, 업스트림이 업데이트될 때마다 다시 해야 한다. 모든 앱이 `bUseViewer` 같은 친절한 플래그를 주지도 않는다. 그래서 **의존성 채우기(Xvfb)** 가 **없애기(주석·플래그)** 보다 값싸다.

## 일반화

**축 A — 엔드포인트가 없으면 대역을 세운다.** 클라이언트는 *프로토콜을 말할 줄 아는 무언가*가 반대편에 있기만 하면 된다. 진짜 본체일 필요는 없다.

- Xvfb = 진짜 모니터 X 서버의 대역
- 코드 테스트의 **in-memory SQLite**(진짜 DB 대신), **mock 서버**(진짜 외부 API 대신) — 똑같은 발상
- 그 반대편 전략은 "의존성을 채우지 말고 없애기"(`bUseViewer:false`) — 채우기 vs 없애기, 둘을 견줘 싼 쪽을 고른다

**축 B — 엔드포인트는 주소로 찾는다.** `DISPLAY=:99`, `localhost:5432`, 환경변수 `DATABASE_URL` — 전부 "어디로 연결할지"를 가리키는 주소표일 뿐이다.

(처음엔 `localhost`를 "가짜 서버 세우기"와 같은 거라 뭉뚱그렸는데, 따져보니 아니었다. `localhost`는 축 B(주소)고, 가짜 서버 세우기는 축 A(대역)다. Xvfb 사건은 두 축을 다 쓴다 — 대역을 세우고(A), `:99`라는 주소로 맞춘다(B).)

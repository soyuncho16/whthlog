## 한 줄

Discord 아이콘을 누르면 `A JavaScript error occurred in the main process`로 죽었다. Chromium 프로세스 싱글톤(파일 락 + 유닉스 도메인 소켓)이 강제 종료로 어긋난 문제였고, 싱글톤 파일을 지워서 풀었다.

## 증상

Discord가 업데이트하라길래 tar를 받아 풀었다. 그런데 아이콘을 누르면 옛 버전이 열렸다.

tar를 수동으로 갈아끼우고 `.deb`로 다시 깔면서 증상이 네 번 바뀌었다.

1. tar를 풀었는데 아이콘은 옛 버전을 연다.
2. `/usr/share/discord`를 새 파일로 덮었는데도 "업데이트하라"가 계속 뜬다.
3. `.deb`로 깔았더니 터미널 `discord`는 되는데 아이콘은 안 열린다.
4. 아이콘을 누르면 `A JavaScript error occurred in the main process` 다이얼로그가 뜬다.

같은 실행 파일인데 터미널과 `gtk-launch`로는 뜨고 아이콘만 죽었다. 프로세스 목록을 보니 Discord 메인 프로세스가 두 개 떠 있었다(`Discord --url --` 가 PID 1212417, 1216778).

## 내 진단

처음 막혔을 때 짚은 건 경로였다.

> **내 진단** 아이콘이 옛 버전을 여는 건 아이콘이 가리키는 것과 내가 새로 푼 파일이 다른 자리에 있어서다. 런처(`.desktop`)의 `Exec`는 `/usr/share/discord/Discord`를 부르는데, 나는 tar를 다운로드 폴더에 풀어놨다. 런처는 포인터지, 파일을 어디 풀든 따라오지 않는다.

이 진단은 맞았지만 그게 다가 아니었다. `/usr/share/discord`를 덮어써도 증상이 안 끝났다. 진짜 원인은 더 아래에 있었고, 미리 떠올린 게 아니라 프로세스 목록을 같이 보다가 나왔다.

## CS 개념

Electron/Chromium 기반 데스크톱 앱은 "한 번에 한 인스턴스"를 지키려고 유저 데이터 폴더(`~/.config/discord`)에 싱글톤 파일 세 개를 둔다.

- `SingletonLock`: `호스트명-PID` 를 가리키는 심볼릭링크. 지금 주인이 어느 PID인지 적는다.
- `SingletonSocket`: `/tmp` 의 유닉스 도메인 소켓을 가리키는 링크. 새로 뜬 프로세스가 기존 주인에게 명령(열 URL 같은)을 넘기는 전화선이다.
- `SingletonCookie`: 같은 인스턴스가 맞는지 확인하는 토큰.

이게 값이 1인 세마포어, 곧 뮤텍스다. 한 놈만 임계영역(앱의 프로필 폴더)에 들어가게 막는 걸 파일로 구현한 것이다.

그 "한 놈만"을 누가 정하느냐가 핵심이다. 따로 검토하는 코드는 없고, OS의 원자적 파일 생성이 그 역할을 한다.

- 한 경로에는 파일 하나만 존재한다. `symlink(타깃, 링크경로)` 시스템콜은 링크경로가 이미 있으면 실패한다(EEXIST). 덮어쓰는 중간 단계가 없다.
- 두 프로세스가 동시에 `symlink(".../SingletonLock")`을 호출해도 커널이 한 놈만 성공시킨다. 성공한 놈이 주인이다.
- 락을 잡는다는 건 `symlink()` 호출이 성공한다는 뜻이다. 이 원자적 동작이 뮤텍스에 필요한 test-and-set이다. 그래서 파일 하나가 자물쇠가 된다.

새 프로세스가 뜨면, 락을 만들려다 EEXIST가 나면 주인이 이미 있다는 걸 안다. 그 주인의 PID가 살아 있으면 `SingletonSocket`으로 연결해 "이 URL 열어줘"를 넘기고 자기는 종료한다. 주인이 죽어 보이면 락을 깨고 자기가 주인이 된다.

이 락과 소켓은 프로세스가 정상 종료할 때 스스로 지운다. 이 정리가 빠지면 다음 실행이 어긋난 상태를 물려받는다.

## 좁혀가기

증상이 바뀔 때마다 다른 데가 원인이었다. 하나씩 짚어봤다.

1. 포인터와 파일(내 진단). 위의 경로 문제. 맞게 고쳤는데도 "업데이트하라"가 계속 떴다.

2. 낡은 파일. 막혀서 Discord 업데이트 API를 조회했다(도구로 확인한 사실). 최신은 1.0.144인데 내가 받은 줄 알았던 tar는 0.0.130, 한참 옛날 빌드였다. 다운로드했다는 것과 받은 게 최신이라는 건 다른 얘기였다. 경로는 맞게 고쳤지만 덮어쓴 원본이 낡았다.

3. 설치 구조 변경. `.deb`로 갈아탔다. 터미널 `discord`는 되는데 아이콘은 안 열리고 `build_info.json`이 어디에도 없었다. `dpkg -L discord`를 보니 새 deb는 앱 본체 대신 `updater_bootstrap`만 깐다. 1.0.x는 부트스트랩이 본체를 유저 프로필(`~/.config/discord/app-1.0.144/`)로 받아 자가 업데이트하는 구조였다. `build_info.json`이 `/usr/share`에 없던 건 잘못 깐 게 아니라 설치 구조가 달라진 탓이었다.

4. 빗나간 가설. 이 시점에 페어로 도와준 쪽에서 "유저레벨 `.desktop`이 시스템 런처를 가린다(XDG 우선순위)"고 의심했는데 틀렸다. `~/.local/share/applications`에는 `discord.desktop`이 없었다.

5. 프로세스 싱글톤(진짜 원인). 답은 프로세스 목록에 있었다. Discord 메인이 둘 떠 있었고(1212417, 1216778), `SingletonLock`은 살아 있는 1216778을 가리켰다. 싱글톤 락과 소켓의 역할은 페어 디버깅에서 듣고 알았고, 그 재료로 시간순으로 엮어봤다.

> A가 생성됨. 락도 소켓도 A를 가리킴. A를 `pkill -9` 함. 제대로 안 죽음. A는 살아 있는데 락과 소켓은 어케 된 건지 모르겠음. B가 생겨남. 락은 그 순간 제대로 된 주인을 못 가리켜서 B를 가리킴. 소켓은 죽은 옛 회선. 나중에 봤더니 A랑 B가 둘 다 살아 있고, 락은 B를 가리키고, 소켓은 아무도 안 받음. 이때 새로 켠 C(아이콘 클릭)가 락을 봤더니 누가 있어서 "소켓으로 넘겨주고 가야지" 했는데, 소켓으로 전하려고 보니 받는 놈이 없음. 실패. 터짐.

타임라인에서 직접 잡은 두 가지가 매듭이다.

터진 건 새로 켠 C다. 기존 주인(B)에게 소켓으로 명령을 넘기려던 C가, 넘길 회선이 죽어서 터졌다.

그리고 타임라인에서 "모르겠음"으로 비워둔 칸. A가 살아 있었다면 규칙상 B는 락을 못 뺏어야 한다. 주인이 죽어 보일 때만 뺏을 수 있으니까. 그런데 B가 가져갔다는 건 그 순간 게이트가 오작동했다는 뜻이고, 그 지점이 `kill -9`이 낸 상처다. 어느 kill이 어느 회선을 죽였는지는 스냅샷 하나로 복원되지 않는다. 억지로 채우면 틀린 얘기가 되니 비워둔다.

## 해결 & 왜 되는가

주인이 없는데 남은 락과 소켓을 지우고, 깨끗한 상태에서 하나만 띄운다.

```bash
pkill -9 -i discord                                   # 두 인스턴스와 자식 전부 종료
rm -f ~/.config/discord/Singleton{Lock,Socket,Cookie} # 어긋난 싱글톤 세 개 제거
# 그 뒤 아이콘으로 실행하면 단일 인스턴스와 새 싱글톤이 생긴다
```

프로세스를 다 죽이면 주인 후보가 사라지고, Singleton 파일을 지우면 다음 실행이 락·소켓·쿠키를 새 세트로 만든다. 그 뒤로는 아이콘을 또 눌러도 새 프로세스가 정상 소켓으로 넘겨 기존 창만 띄운다. 상호배제가 돌아온 것이다.

이 파일을 지워도 안전한 이유도 같다. Singleton 세 개에는 메시지나 로그인, 설정 같은 데이터가 없다. 그건 `Cookies`, `Local Storage` 같은 다른 파일에 있고 건드리지 않았다. 싱글톤 파일은 지금 누가 도는지만 적는 장부라, 앱이 꺼진 상태에서 지우면 다음 실행 때 다시 생긴다.

강제 종료가 문제를 만든 이유. `kill -9`(SIGKILL)은 프로세스에 정리할 틈을 안 준다. 정상 종료(SIGTERM 핸들러나 앱의 Quit)는 끝나기 직전에 Singleton 파일을 지우는데, SIGKILL은 그걸 건너뛰어 락과 소켓을 어긋난 채 남긴다. 안 되면 `kill -9`으로 강제 종료하는 습관이 다음 실행을 깨뜨린다. 제대로 끄려면 창에서 Quit(Ctrl+Q)을 쓰고, 어쩔 수 없이 강제 종료했으면 Singleton 파일도 같이 지운다.

## 일반화

락은 파일이라 프로세스가 죽어도 자동으로 안 지워진다. 정상 종료가 그걸 지우는데, 그 정리가 빠지면 낡은 락이 남아 다음 시작을 막는다. 같은 패턴이 여기저기 있다.

- X 서버 락 `/tmp/.X0-lock` (죽은 X가 남기면 다음 X가 못 뜸)
- `apt`/`dpkg` 락 `/var/lib/dpkg/lock` (강제 종료 후 `could not get lock`)
- PostgreSQL `postmaster.pid`, Redis 같은 데몬의 PID 파일

같은 발상이 한 PC를 넘어 클러스터로도 간다. 여러 서버가 한 서비스처럼 도는데 어떤 작업은 한 대만 해야 할 때(정산 메일 같은), 공유 저장소(Redis, Zookeeper)에 락을 잡는다. 공유 표식이 로컬 파일 대신 네트워크 너머의 키일 뿐, 원자적으로 한 놈만 잡는다는 본질은 같다. 이게 분산락이다.

증상은 믿을 게 못 된다. 아이콘이 옛 버전, 업데이트 알림, 안 열림, JS 에러로 네 번 바뀌는 동안 진짜 원인은 한 곳, 프로세스 싱글톤이었다. 런처와 설치 파일과 패키지 장부와 실행 중 프로세스는 서로 다른 출처라, 넷이 다 맞아야 정상이다. 한 군데서 답이 안 나오면 다른 출처를 의심한다.
